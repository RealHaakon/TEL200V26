\section{Introduction}
\subsection{Yumi Application}
The goal for the yumi application part of the project was to enable the robot to manipulate objects.
There are three blocks that shall be moved to their appropriate location, a button input for each block,
and one emergency stop button. The blocks are all different shapes, and each have two spots on the table.
The corresponding button input shall cause the robot to pick the block up and move it to its other spot.
It should stop in its tracks at the push of the emergency stop button. There is also a fourth button that
will move the robot back to its home position. To achieve this goal we will program the yumi lab robot
using Robot Studio.

\subsection{Yumi Challenge}
The goal for the yumi challenge part of the project was to enable the robot to play tic tac toe.
By placing paper on the table and making the robot grip a marker we could program it to draw shapes in
the locations we want. Specifically, we made it draw an X at coordinate points in a $3\times3$ coordinate field.
It's controlled through button inputs, one for the $x$-coordinate and one for the $y$-coordinate. Pressing the
button will incrementally increase the corresponding value by 1. A third button is used to indicate that the
desired values are reached, and start the related motion. To achieve this goal we will program the yumi lab robot
using Robot Studio.

\section{Method}

\subsection{YuMi Application}
The control software for the dual-arm YuMi robot (IRB 14000) was developed in ABB's RAPID programming language using RobotStudio. The application implements a modular pick-and-place task that handles three distinct geometric objects --- cube, cylinder, and triangular prism --- triggered by external digital input signals. A central safety and recovery mechanism is provided through interrupt-driven emergency stop handling.

The program is contained in \texttt{MODULE Module1}. It defines a large number of \texttt{CONST robtarget} constants that store the taught positions for each object. These include safe approach points, gripping poses, lifting positions, transport poses, and drop-off locations. Each target is associated with the appropriate work object (\texttt{wobj}) --- \texttt{Kube}, \texttt{sylinder}, \texttt{prisme} for pick-up phases, and typically \texttt{wobj0} for home and global movements. The positions were taught using the FlexPendant or RobotStudio's interface and include complete pose data (Cartesian position, orientation quaternion, robot configuration, and external axes values).

The main entry point is the \texttt{Main()} procedure, which first initializes the gripper system by calling \texttt{g\_Init}. To avoid the common runtime error \texttt{ERR\_ALRDYCNT} caused by connecting the same interrupt multiple times, an initialization guard is used:

\begin{verbatim}
IF NOT interrupts_setup THEN
    SkipWarn;
    CONNECT stop_emer WITH EmergencyStop;
    ISignalDI di_EmergencySituation, high, stop_emer;
    interrupts_setup := TRUE;
ENDIF
\end{verbatim}

The program then enters an infinite loop that continuously polls three digital input signals (\texttt{di\_cube}, \texttt{di\_cylinder}, \texttt{di\_prism}) and calls the corresponding handling procedure when a signal is high:

\begin{verbatim}
WHILE TRUE DO
    IF di_cube = 1 THEN Cube;
    ENDIF
    IF di_cylinder = 1 THEN syl;
    ENDIF
    IF di_prism = 1 THEN prism;
    ENDIF
ENDWHILE
\end{verbatim}

Each object-specific procedure (\texttt{Cube}, \texttt{syl}, \texttt{prism}) follows a similar pick-and-place sequence:
\begin{enumerate}
    \item Move to home position
    \item Approach the pick-up area
    \item Open gripper (\texttt{g\_GripOut}) and wait
    \item Descend to gripping pose
    \item Close gripper (\texttt{g\_GripIn}) and wait
    \item Lift back to approach height
    \item Transport to drop area
    \item Descend to release pose
    \item Open gripper and wait
    \item Lift slightly
    \item Return to home position
\end{enumerate}

Most motions use \texttt{MoveJ} with speed \texttt{v500}  and accuracy \texttt{fine} at critical points or \texttt{z100} during faster transfers. 
Wait times of 2 seconds after gripper commands ensure reliable mechanical action.

The emergency stop is implemented as a trap routine triggered on the rising edge of \texttt{di\_EmergencySituation}:

\begin{verbatim}
TRAP EmergencyStop
    SkipWarn;
    StopMove;
    WaitUntil di_EmergencySituation = 0;
    StorePath;
    IF DInput(di_home) = 1 THEN
        ClearPath;
        MoveJ home, v500, fine, Servo\WObj:=wobj0;
        WaitTime 1;
        ExitCycle;
    ELSE
        RestoPath;
        StartMove;
    ENDIF
ENDTRAP
\end{verbatim}
When the emergency input becomes active, motion stops immediately (\texttt{StopMove}). 
After the signal is released, the program checks \texttt{di\_home}. 
If high, it clears the path context, moves to the safe \texttt{home} position, 
and executes \texttt{ExitCycle} to restart from the beginning of \texttt{Main()}. 
Otherwise it restores and resumes the interrupted path. This mechanism provides both immediate safety response and flexible recovery.

\subsection{YuMi Tic-Tac-Toe Challenge}
A tic-tac-toe demonstration was developed for the dual-arm YuMi robot to play against a human opponent. The robot exclusively draws ``X'' symbols, while the human manually places ``O'' symbols on a physical 3×3 grid. The human communicates their move to the robot using external digital inputs.
\begin{figure}[H]
        \centering
        \includegraphics[width=.7\textwidth]{images/yumichallengeSetup.png}
        \caption{yumi Challenge setup}
        \label{fig:yumi Challenge setup}
    \end{figure}

The robot uses two digital inputs to select the grid cell:
- \texttt{di\_x} increments the X coordinate
- \texttt{di\_y} increments the Y coordinate
- \texttt{di\_exe} triggers execution of the selected cell

A custom CAD model was designed and 3D-printed for both the pen (marker) and the pen holder to ensure reliable and repeatable tool pickup and placement.

The nine grid cells are divided between the two arms according to the following mapping (first digit = X coordinate, second digit = Y coordinate):

\begin{table}[H]
\centering
\caption{Grid cell assignment to robot arms}
\begin{tabular}{c|c|c|c}
\toprule
\textbf{} & \textbf{X=0} & \textbf{X=1} & \textbf{X=2} \\
\midrule
\textbf{Y=0} & Right: P00 & Right: P10 & Left: P20 \\
\textbf{Y=1} & Right: P01 & Right: P11 & Left: P21 \\
\textbf{Y=2} & Right: P02 & Left: P12 & Left: P22 \\
\bottomrule
\end{tabular}
\end{table}

Each path procedure follows a simple sequence:
\begin{itemize}
    \item Move to home position
    \item Pick the pen from holder
    \item Move to cell-specific waypoints and draw the X
    \item Return pen to holder
    \item Return to home and resets X and
\end{itemize}

Motion commands use \texttt{MoveJ} at speed \texttt{v500}, with \texttt{fine} or small zones (\texttt{z50}--\texttt{z100}) for smooth and accurate drawing. Gripper commands are followed by \texttt{WaitTime 2} to ensure stable handling.

The same interrupt-based emergency stop system is implemented as in the pick-and-place application. For correct operation, the controller must be set to \textbf{continuous} run mode.

This project demonstrates YuMi's precision, tool handling capability, and safe interaction with humans in a fun, educational game application.

\section{Discussion}
We started work on making the robot autonomously respond to user moves. By creating a python script to function
as a tic-tac-toe AI, as well as a python script to serve as a bridge between the AI and the robot with TCP/IP, the
robot can theoretically play without being manually controlled by the human. We envisioned the following logical flow: \\\\
$\text{User input for user move} \rightarrow \text{Robot sends the user input as a packet to the python server} \rightarrow
\text{Python server calls the AI to get the robots move} \rightarrow \text{Server returns robot move packet to robot} \rightarrow
\text{Robot follows the appropriate path for the received xy-coordinates}$ \\\\
With this structure an autonomous robot tic-tac-toe player can be achieved, though it requires the robot
to be connected via ethernet.
