\section{Introduction}
\subsection{Yumi Application}
The goal for the yumi application part of the project was to enable the robot to manipulate objects.
There are three blocks that shall be moved to their appropriate location, a button input for each block,
and one emergency stop button. The blocks are all different shapes, and each have two spots on the table.
The corresponding button input shall cause the robot to pick the block up and move it to its other spot.
It should stop in its tracks at the push of the emergency stop button. There is also a fourth button that
will move the robot back to its home position. To achieve this goal we will program the yumi lab robot
using Robot Studio.

\subsection{Yumi Challenge}
The goal for the yumi challenge part of the project was to enable the robot to play tic tac toe.
By placing paper on the table and making the robot grip a marker we could program it to draw shapes in
the locations we want. Specifically, we made it draw an X at coordinate points in a $3\times3$ coordinate field.
It's controlled through button inputs, one for the $x$-coordinate and one for the $y$-coordinate. Pressing the
button will incrementally increase the corresponding value by 1. A third button is used to indicate that the
desired values are reached, and start the related motion. To achieve this goal we will program the yumi lab robot
using Robot Studio.
\\\\
Precision movements like these are very relevant in the current landscape of robotics. While the use case in industry
might not be to match shapes like this, the ability to precisely manipulate objects with care and accuracy
is incredibly important for areas like manufacturing. It allows for the handling of objects too
difficult for people to handle with appropriate care, and it allows for automation of
repetitive tasks.

\section{Method}

\subsection{YuMi Application}
The control software for the dual‑arm YuMi robot (IRB 14000) was developed in ABB’s RAPID language using RobotStudio. The application implements a pick‑and‑place task for three objects (cube, cylinder, and triangular prism), each triggered by a dedicated digital input signal. The robot moves each object from its original position on the table to a corresponding “new” position, while returning to a safe home pose after each sequence.

The program is contained in \texttt{MODULE Module1} and defines a set of \texttt{CONST robtarget} positions for each object, including approach points, gripping poses, and drop‑off locations. These targets are associated with separate work objects (\texttt{Kube}, \texttt{sylinder}, \texttt{prisme}), while global movements use \texttt{wobj0}. All motions are executed using \texttt{MoveJ} at speed \texttt{v500} with \texttt{fine} or \texttt{z100} accuracy, ensuring smooth and repeatable trajectories. Gripper commands (\texttt{g\_GripOut}/\texttt{g\_GripIn}) are followed by \texttt{WaitTime 2} to ensure reliable mechanical action.

The system includes a fully functional emergency stop, implemented via an interrupt on the signal \texttt{di\_EmergencySituation}. When activated, the robot halts immediately using \texttt{StopMove} and the output \texttt{do\_emergency} is set high. After the emergency button is released, the robot checks the \texttt{di\_home} signal; if active, it returns to the \texttt{home} position and restarts the cycle, otherwise it restores and resumes the interrupted path. The emergency logic is supported by ABB Smart Components in RobotStudio, which provide a simulated realistic button and emergency circuit behavior for testing and validation.

\subsection{YuMi Tic-Tac-Toe Challenge}
The Tic-Tac-Toe project was implemented on the dual-arm YuMi robot using two independent RAPID modules --- one for the left arm and one for the right arm. Each arm operates autonomously but follows the same general logic: to select and draw an ``X'' in one of nine grid cells based on incoming digital signals.

\subsubsection*{Program structure}
Both arms continuously poll three digital inputs:
\begin{itemize}
  \item \texttt{di\_cube}: increments the X coordinate
  \item \texttt{di\_cylinder}: increments the Y coordinate
  \item \texttt{di\_prism}: confirms path execution
\end{itemize}
Two counters (\texttt{X} and \texttt{Y}) track the selected grid cell. Depending on the X--Y combination, the program calls one of several path procedures such as \texttt{Path\_P00}, \texttt{Path\_P11}, or \texttt{Path\_P22}.

The human-robot grid is divided between the arms as follows:

\subsubsection*{Path execution}
When a cell is selected, the corresponding path procedure executes a fixed sequence:
\begin{enumerate}
  \item Move to the home position.
  \item Move to \texttt{PenCenter} (safe pose above pen holder).
  \item Open gripper (\texttt{g\_GripOut}), descend to \texttt{PenPickDrop}, and close gripper (\texttt{g\_GripIn}) to pick up the pen.
  \item Move through a series of target points to draw an ``X''.
  \item Return to \texttt{PenCenter}, drop off the pen, and return to home.
  \item Reset both \texttt{X} and \texttt{Y} to zero, ready for the next move.
\end{enumerate}
Joint motions (\texttt{MoveJ}) are used for most positioning, while linear motions (\texttt{MoveL}) ensure smooth, straight drawing lines. Speeds are set to \texttt{v500} with fine or small blending zones (\texttt{z50}--\texttt{z100}) depending on path precision.

\subsection{CAD Models}
To support reliable pen handling, a custom CAD design was created for both the \textbf{pen holder} and the \textbf{pen adapter}. The parts were designed in Autodesk Fusion 360 and 3D-printed in PETG.
The holder fixes the pen in a repeatable position on the table, while the adapter ensures proper orientation and grip alignment with the robot's fingers. This modular approach improved drawing consistency and made tool pickup or drop-off fully repeatable between runs.
Also a pen was designed in Autodesk that is identical in dimentions to the marker used in order to get correct point to floor distance while programming in rapid.

\section{Results}

\subsection{Yumi application}
The YuMi application was successfully implemented and verified both in simulation and on the physical robot, with the pick-and-place sequences for all three objects executing as planned. A key factor in achieving robust behavior was the consistent use of joint-space motion commands (\texttt{MoveJ}), which ensured smooth trajectories and reduced the risk of encountering configurations analogous to gimbal lock in the robots wrist by avoiding problematic intermediate orientations.
As a result, the robot was able to move reliably between home, approach, and grasp poses without unexpected singular behavior or loss of orientational degrees of freedom.

\subsection{Yumi challenge}
In the Tic-Tac-Toe challenge, the overall logic for coordinate selection and path execution functioned as intended, but the live experiments revealed several practical limitations. During repeated pen handover and parking operations, we experienced collisions between the pen and the holder, which eventually led to mechanical failure of one gripper finger,
despite using nominally identical target points and trajectories for the right and left arms.
The YuMi system proved very sensitive to external impacts and contact forces; even small unintended forces triggered the built-in collision detection,
causing the robot to stop and requiring manual intervention before motion could resume.

The drawing performance was also somewhat inconsistent, with small deviations in the contact point or pen orientation leading to visible variation in line quality and stroke placement, which highlighted the need for highly accurate and repeatable target positions on the paper surface.
The custom-designed pen holder and gripper adapter were therefore a significant asset,
as they helped standardize the pen pose relative to the robot and improved the consistency between simulation and live execution by providing a repeatable mechanical interface during pickup and drop-off.

\section{Discussion}
We started work on making the robot autonomously respond to user moves. By creating a python script to function
as a tic-tac-toe AI, as well as a python script to serve as a bridge between the AI and the robot with TCP/IP, the
robot can theoretically play without being manually controlled by the human. We envisioned the following logical flow: \\\\
$\text{User input for user move} \rightarrow \text{Robot sends the user input as a packet to the python server} \rightarrow
\text{Python server calls the AI to get the robots move} \rightarrow \text{Server returns robot move packet to robot} \rightarrow
\text{Robot follows the appropriate path for the received xy-coordinates}$ \\\\
With this structure an autonomous robot tic-tac-toe player can be achieved, though it requires the robot
to be connected via ethernet.
